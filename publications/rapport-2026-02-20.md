### Hooks

**Voorstel:** Implementeer een basis hook-configuratie met vier bash hooks voor autonome werking
**Bron:** earezki.com — [Claude Code Runs Autonomously with 4 Bash Hooks](https://earezki.com/ai-news/2026-02-14-4-hooks-that-let-claude-code-run-autonomously-with-zero-babysitting/)
**Type:** fundamenteel
**Toelichting:** Hooks geven deterministische controle over elke fase van Claude Code's lifecycle (PreToolUse, PostToolUse, etc.). In plaats van reactief toezicht te houden, kun je met vier gerichte bash hooks een autonoom besluitnemingssysteem opzetten. Dit verandert de manier waarop je met Claude Code werkt fundamenteel: van handmatig goedkeuren naar gedefinieerde grenzen waarbinnen de agent vrij opereert.

```yaml
# In .claude/settings.json — voorbeeldstructuur
hooks:
  PreToolUse:
    - matcher: "Write|Edit"
      command: "bash .claude/hooks/pre-write-check.sh"
  PostToolUse:
    - matcher: "Bash"
      command: "bash .claude/hooks/post-bash-validate.sh"
  Notification:
    - command: "bash .claude/hooks/notify-complete.sh"
```

### Contextmanagement

**Voorstel:** Hanteer de 75%-regel: start een nieuwe sessie vóór 80% contextgebruik
**Bron:** claudefa.st — [Claude Code Context Window: Optimize Your Token Usage](https://claudefa.st/blog/guide/mechanics/context-management)
**Type:** klein
**Toelichting:** Claude's 200K token context window is actief werkgeheugen dat meetbaar slechter functioneert naarmate het voller wordt. Auto-compaction triggert nu al bij 75% (eerder 90%+), wat ~50K tokens vrijhoudt voor redenering. Praktische regel: monitor je contextgebruik en gebruik `/compact` proactief, of start een nieuwe sessie bij 80%.

## Paradigma-check

### 1. "Approval fatigue" ondermijnt het huidige permissiemodel

De beveiligingsartikelen van deze week stellen de standaard permissiemodus ter discussie. Het argument: herhaalde toestemmingsdialogen trainen gebruikers om reflexmatig "ja" te klikken, waardoor het beveiligingsmodel zinloos wordt. De oplossing is niet "meer dialogen" maar **sandboxing + gedefinieerde grenzen** — een fundamenteel andere benadering dan het huidige click-to-approve model.

**Implicatie:** Als je nu op handmatige goedkeuring vertrouwt als beveiligingsmaatregel, is dat mogelijk schijnveiligheid. Overweeg sandboxing als structurele oplossing.

### 2. Van hiërarchische subagents naar peer-to-peer Agent Teams

Het opkomende Agent Teams-model (peers die direct communiceren) verschilt fundamenteel van het subagent-model (orchestrator die taken delegeert). Dit suggereert dat de toekomst van multi-agent workflows niet één controller met workers is, maar een team van gelijkwaardige specialisten. Dit is nog experimenteel, maar het is de moeite waard om je mentale model bij te stellen.

### 3. Claude Code is geen app-generator

Het artikel van Hack Celeration benadrukt dat Claude Code uitblinkt in het **transformeren** van bestaande code, maar faalt bij het genereren van complete applicatie-architecturen uit een enkele prompt. State management, complexe interacties en architecturale samenhang worden gemist. De les: gebruik Claude Code als een krachtige medewerker die werkt binnen door jou gedefinieerde kaders, niet als een autonome architect.

**Bron:** [Claude Code Review 2026](https://hackceleration.com/claude-code-review/) (Hack Celeration)

### 4. Context beschermen is belangrijker dan context vullen

De verschuiving van compaction bij 90% naar 75% (waardoor 50K tokens vrijblijven voor redenering) illustreert een paradigmaverschuiving: het gaat niet om zoveel mogelijk context laden, maar om **ruimte bewaken voor redenering**. Een halfvolle context met goede redeneercapaciteit levert betere resultaten dan een volle context waar het model nauwelijks kan nadenken.

**Bron:** [How Claude Code Got Better by Protecting More Context](https://hyperdev.matsuoka.com/p/how-claude-code-got-better-by-protecting)
